## 4. Root/admin layout separation

Сделано разделение на два уровня layout:

- корневой `src/app/layout.tsx` теперь рендерит только `html/body`;
- публичный сайт обёрнут в `src/app/(site)/layout.tsx`, где подключён `MainLayout`;
- `src/app/admin/...` остался вне `(site)`, поэтому admin-страницы больше не наследуют публичный Header/Footer.

Результат:

- `/`, `/news`, `/news/[slug]`, `/gia-9`, `/gia-11`, `/kachestvo-obrazovaniya`, `/o-centre` и другие публичные маршруты продолжают жить внутри `MainLayout`;
- `/admin`, `/admin/login`, `/admin/news`, `/admin/users` и другие admin routes рендерятся без публичной шапки и футера.

Ручная HTTP-проверка:

- `GET http://localhost:3002/admin/login` вернул `200`;
- в ответе не найдено публичных ссылок `Главная` и `Контакты`;
- в ответе есть admin-контент.

## 6. Scheduled publishing logic

### Backend-логика

В `NewsService` приведена к явной схеме публикации:

- черновик:
  - `isPublished = false`
- запланировано:
  - `isPublished = true`
  - `publishedAt > now`
- опубликовано:
  - `isPublished = true`
  - `publishedAt <= now`

Исправлено публичное поведение:

- `GET /api/public/news`
- `GET /api/public/news/:slug`

теперь возвращают новости только если:

- `is_published = true`
- `published_at <= now`
- `deleted_at = null`

В admin DTO добавлено вычисляемое поле:

- `status: "draft" | "scheduled" | "published"`

### Frontend-логика

В форме новости добавлен режим публикации:

- `draft`
- `publish-now`
- `schedule`

Mapping:

- `draft` -> `isPublished=false`
- `publish-now` -> `isPublished=true`, `publishedAt=now`
- `schedule` -> `isPublished=true`, `publishedAt=selected datetime`

Для `schedule` поле даты обязательно.

В admin news list теперь используется `status`, а не только `isPublished`.

### Ручная проверка

Через admin API создана тестовая новость:

- slug: `scheduled-manual-1783605490`
- title: `Scheduled Manual News`
- status: `scheduled`
- publishedAt: future datetime

Проверено:

- backend create response вернул `status = scheduled`;
- публичная страница `/news` не содержит эту новость до момента публикации.

Дополнительно бизнес-логика закрыта e2e-тестами.

### Что делает `POST /api/user/admin/users`

Доступ:

- только `super-admin`

Request DTO:

- `name`
- `email`
- `password`
- `isSuperAdmin`
- `sectionIds`

Поведение:

- валидирует email;
- хеширует пароль через `bcryptjs`;
- не возвращает password/hash наружу;
- проверяет существование переданных `sectionIds`;
- создаёт пользователя;
- назначает permissions;
- возвращает safe DTO:
  - `id`
  - `name`
  - `email`
  - `isSuperAdmin`
  - `permissions`

Обработка ошибок:

- дубль email -> `EMAIL_ALREADY_IN_USE`

### Sections endpoint

Добавлен/используется:

- `GET /api/user/admin/sections`

Response содержит:

- `id`
- `sectionId`
- `title`
- `kind`
- `route`
- `parentId`
- `order`
- `isEditable`

## 14. Ошибки и исправления

### 14.1. Prisma DLL lock на Windows

Проблема:

- `npm run test:e2e` и `prisma generate` периодически падали с:
  - `EPERM: operation not permitted, unlink ... query_engine-windows.dll.node`

Причина:

- запущенный backend/node watcher или production node процесс держал Prisma engine file.

Исправление:

- останавливался активный backend node-процесс;
- после этого `npm run test:e2e` проходил успешно.

### 14.2. Сырой `Bad Request` на admin login

Ранее фронт показывал сырое сообщение `Bad Request`.

Исправление:

- в `src/shared/admin/api-error.ts` добавлена нормализация backend error codes в человекочитаемые сообщения;
- для `AUTH_FAIL` теперь используется сообщение про неверный email или пароль.

### 14.3. `/admin` был внутри публичного layout

Исправление:

- публичные страницы перенесены в `(site)`;
- admin оставлен отдельной веткой app router.

## 15. Риски

### 15.1. UX перехода обычного admin на `/admin/users`

Backend RBAC работает корректно, но в ручной HTTP-проверке route вёл себя не идеально прозрачно:

- доступ запрещён;
- при этом наблюдался редирект, который нужно ещё один раз проверить именно в браузере, чтобы убедиться, что пользователь реально видит expected forbidden screen, а не login screen.

### 15.2. Проверка select категорий без полноценного UI automation

Создание categories подтверждено, но наличие конкретной новой рубрики в client-side select формы новости стоит дополнительно один раз проверить интерактивно в браузере.

### 15.3. В shared API client осталась старая localStorage-ветка

В `D:\Desktop\dev\web\orders\kciasso-frontend\src\shared\api\client.ts` всё ещё есть legacy-чтение токена из:

- `window.localStorage.getItem("vibe_admin_access_token")`

Текущий admin flow на это не опирается, потому что использует server-side cookie и explicit `Authorization` в adapters, но этот legacy-хвост лучше убрать отдельно, чтобы не оставлять смешанную auth-модель.
