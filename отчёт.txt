Дата: 2026-07-16.

## 1. Цель

Найти фактические причины текущих frontend/backend CI/deploy ошибок, проверить реальные ветки, workflows, lockfile, Docker build context, production image и seed, внести минимальные исправления, проверить их в Linux/Node 22, подтвердить remote commits и результат GitHub Actions/Dokploy trigger.

## 3. Подтверждённые причины

### Frontend

Причина не в различии локального и CI npm: старый remote lock действительно был несогласован с `package.json`. Его чистый запуск в `node:22-bookworm-slim` с Node `22.23.1` и npm `10.9.8` воспроизвёл ровно четыре CI ошибки:

- `@emnapi/runtime@1.11.2`;
- `@emnapi/core@1.11.2`;
- `@emnapi/core@1.11.1`;
- `@emnapi/runtime@1.11.1`.

Исправленный staged lock в том же окружении установил 705 пакетов без EUSAGE. `@emnapi/*` приходит через optional WASM/NAPI runtime dependency chain (`@napi-rs/wasm-runtime`, Rolldown/unrs bindings). `.npmrc`, workspaces и дополнительные lockfiles отсутствуют.

Прошлый push ничего не изменил, потому что опечатка `git coomit` не создала commit, а обычный push передаёт только commits, не staged working tree.

### Backend

Seed выполняется в runner через `prisma db seed`/`ts-node`, а production runner не содержит `src/**`. Это прямое несоответствие runtime image и seed import graph. Проверка всех `prisma/seed*.ts` показала единственную зависимость seed от `src/**` — site settings constants. Копирование одного source-файла создало бы хрупкую частичную runtime-зависимость; вместо этого seed сделан самодостаточным.

Seed запускается при `RUN_SEED_ON_BOOT=true`, потому что он идемпотентно upsert-ит обязательные sections, categories, news и site settings, а safe super-admin path не меняет пароль существующего пользователя. Повторный seed на той же тестовой базе прошёл успешно.

## 8. Docker/Dokploy и что перезапускать

Оба успешных workflow уже опубликовали SHA-tagged/latest GHCR images и успешно вызвали Dokploy `application.deploy`. Поэтому повторный GitHub Actions rerun не требуется.

Если Dokploy UI всё ещё показывает старые containers, вручную redeploy/recreate нужно сделать для обеих applications:

- frontend image/commit `3a5d78bcf3a22a66b87ad58b407296c721e6f399`;
- backend image/commit `588f2b74770291af5fd9ab5cb17416cb9dcf138e`.

Dokploy application-side branch/settings не были доступны без его API/UI credentials. Однако фактический deploy path подтверждён workflow: build идёт из `master`, image публикуется, затем Dokploy deploy endpoint успешно вызывается.

## 10. Итоговый status и риски

После remote verification оба code trees совпадали с `origin/master` и были clean. Канонические `PROJECT_STATE.md`/`отчёт.txt` сохраняются отдельным documentation commit с `[skip ci]`; production deploy code остаётся в подтверждённом commit `588f2b7`, а обновление отчёта не запускает лишний image deploy.

Оставшиеся риски:

- backend GitHub default branch/remote HEAD всё ещё `main`, хотя deploy branch и local branch — `master`; настройку default branch следует отдельно переключить на `master`, не удаляя legacy `main` без решения владельца;
- фактическую версию запущенных containers в Dokploy UI нельзя независимо прочитать без credentials;
- frontend build использует fallback, когда backend недоступен во время prerender; это существующее предусмотренное поведение, не причина текущего CI сбоя.
